## 2. P2-01: LANGGRAPH ORCHESTRATION

### 2.1 Objective
Replace hand-written `main.py` control flow with a LangGraph `StateGraph`. The `decision_path` field in the trace must be richer (each node appended). All other trace fields remain structurally unchanged.

### 2.2 File Targets

| File | Action |
|---|---|
| `main.py` | Deprecate control flow logic. Keep as thin entrypoint that instantiates and invokes the graph. |
| `graph.py` | Create. Owns `StateGraph` definition, node registration, edge wiring. |
| `state.py` | Create or modify. Defines `AgentState` TypedDict used across all nodes. |
| `nodes/` | Create directory. One file per node. |

### 2.3 AgentState Schema (TypedDict)

```python
class AgentState(TypedDict):
    prompt: str                      # Raw user input
    intent: str                      # Classified intent (set by intent_node)
    context: list[dict]              # Retrieved chunks from PageIndex (set by retrieval_node)
    local_response: str              # Raw output from Phi-3 Mini
    validated: bool                  # Set by validator_node
    validation_record: dict          # Populated by validator_node (see P2-02)
    tool_calls: list[dict]           # Populated by tool_node (see P2-03)
    fallback_triggered: bool         # Set True if validator fails
    fallback_record: dict            # Populated by fallback_node (see P2-04)
    final_response: str              # Output surfaced to caller
    decision_path: list[str]         # Append node name on every node entry
    step_count: int                  # Incremented each reasoning cycle; capped at MAX_STEPS
    plan: dict | None                # Populated by planning_agent (see P2-06); None if not invoked
```

All keys must be present at graph init. Initialise `None`/`[]`/`False`/`0` as appropriate. Never add keys mid-run.

### 2.4 Nodes (register in this order)

| Node Name | File | Responsibility |
|---|---|---|
| `intent_node` | `nodes/intent_node.py` | Classify prompt intent into one of: `plan`, `query`, `tool_call`, `fallback_direct`. Set `state["intent"]`. |
| `retrieval_node` | `nodes/retrieval_node.py` | Query PageIndex with `state["prompt"]`. Populate `state["context"]`. If no hits, set `context = []`. |
| `local_inference_node` | `nodes/local_inference_node.py` | Run Phi-3 Mini. Set `state["local_response"]`. |
| `validator_node` | `nodes/validator_node.py` | Validate `local_response`. Populate `validation_record`. Set `validated`. See P2-02. |
| `tool_node` | `nodes/tool_node.py` | Execute OData stub if `intent == "tool_call"`. Populate `tool_calls`. See P2-03. |
| `fallback_node` | `nodes/fallback_node.py` | Invoke Claude Opus 4.6 if `validated == False`. Populate `fallback_record`. See P2-04. |
| `planning_node` | `nodes/planning_node.py` | Invoke Planning Agent if `intent == "plan"`. Populate `state["plan"]`. See P2-06. |
| `response_node` | `nodes/response_node.py` | Assemble `final_response` from validated local or fallback output. |

### 2.5 Edge Wiring

```
START
  └─► intent_node
        ├─[intent == "plan"]──────────────► planning_node ──► response_node ──► END
        ├─[intent == "tool_call"]─────────► retrieval_node ──► tool_node ──► local_inference_node ──► validator_node
        ├─[intent == "query"]─────────────► retrieval_node ──► local_inference_node ──► validator_node
        └─[intent == "fallback_direct"]───► fallback_node ──► response_node ──► END

validator_node
  ├─[validated == True]────► response_node ──► END
  └─[validated == False]───► fallback_node ──► response_node ──► END
```

### 2.6 Step Counter & Loop Guard

- Increment `state["step_count"]` at the top of `local_inference_node` and `fallback_node`.
- If `step_count >= MAX_STEPS` at entry to any node (except `response_node`), immediately route to `response_node` with `final_response = "MAX_STEPS_EXCEEDED"`.
- `MAX_STEPS = 3` is a module-level constant in `graph.py`. Do not hardcode inline.

### 2.7 `decision_path` Instrumentation

Every node must append its own name to `state["decision_path"]` as its **first line of logic**, before any other operation:

```python
state["decision_path"].append("intent_node")  # example
```

## 3. P2-02: VALIDATOR NODE

### 3.1 Objective
Populate `ValidationRecord` in `state["validation_record"]` and set `state["validated"]` (bool). The validator is deterministic — no LLM calls inside this node.

### 3.2 ValidationRecord Schema

```python
class ValidationRecord(TypedDict):
    verdict: Literal["pass", "fail", "skip"]
    confidence: float          # 0.0–1.0, derived from rule scores
    failure_type: str | None   # None if verdict == "pass"; see failure types below
    rules_applied: list[str]   # Names of rules evaluated
    timestamp: str             # ISO 8601 UTC
```

### 3.3 Validation Rules (implement all)

| Rule Name | Logic | Weight |
|---|---|---|
| `non_empty` | `local_response` is not empty string or whitespace-only | 0.30 |
| `min_length` | `len(local_response.strip()) >= 20` | 0.15 |
| `no_refusal_pattern` | Does not match regex: `r"(I cannot|I'm unable|I don't know|As an AI)"` (case-insensitive) | 0.25 |
| `context_grounded` | If `context` is non-empty, at least one token from top-3 context chunks appears in `local_response` (token overlap > 0) | 0.20 |
| `no_hallucination_signal` | Does not contain phrases: `"as of my knowledge cutoff"`, `"I was trained on"` | 0.10 |

**Confidence** = sum of weights of passing rules.

**Verdict logic:**
- `confidence >= 0.70` → `"pass"`, `validated = True`
- `confidence < 0.70` → `"fail"`, `validated = False`
- `intent == "fallback_direct"` → `"skip"`, `validated = False` (routes to fallback without running rules)

### 3.4 Failure Types

| Failure Type | Trigger Condition |
|---|---|
| `EMPTY_RESPONSE` | `non_empty` fails |
| `BELOW_MIN_LENGTH` | `min_length` fails |
| `REFUSAL_DETECTED` | `no_refusal_pattern` fails |
| `UNGROUNDED` | `context_grounded` fails |
| `HALLUCINATION_SIGNAL` | `no_hallucination_signal` fails |
| `MULTI_FAILURE` | More than one rule fails |
| `SKIPPED` | Validator skipped (fallback_direct intent) |

Set `failure_type` to the first applicable value in the order above; if multiple rules fail, set `MULTI_FAILURE`.

### 3.5 File: `nodes/validator_node.py`

Must export a single callable: `def validator_node(state: AgentState) -> AgentState`

## 4. P2-03: MCP-SHAPED TOOL STUB (OData Read)

### 4.1 Objective
Implement a stub OData read tool that conforms to MCP tool interface shape. Populates `state["tool_calls"]`. Real OData endpoint is not required — stub returns synthetic data. The interface must be wired so a real OData URL can be dropped in with zero structural changes.

### 4.2 Tool Interface Shape

```python
class MCPToolCall(TypedDict):
    tool_name: str          # e.g. "odata_read"
    input: dict             # Parameters passed to tool
    output: dict            # Response from tool
    status: Literal["success", "error", "stub"]
    latency_ms: int         # Simulated or real
    timestamp: str          # ISO 8601 UTC
```

Append one `MCPToolCall` dict to `state["tool_calls"]` per tool invocation.

### 4.3 OData Read Stub Spec

**File:** `tools/odata_read.py`

**Function signature:**
```python
def odata_read(
    entity_set: str,
    filter_expr: str | None = None,
    select_fields: list[str] | None = None,
    top: int = 10
) -> dict:
```

**Stub behaviour:**
- If `ODATA_ENDPOINT` env var is set and non-empty → attempt real HTTP GET to `{ODATA_ENDPOINT}/{entity_set}` with `$filter`, `$select`, `$top` params. Return parsed JSON `value` array.
- If `ODATA_ENDPOINT` is not set → return stub response:
  ```python
  {
      "value": [
          {"id": "STUB-001", "name": "Stub Record A", "entity_set": entity_set},
          {"id": "STUB-002", "name": "Stub Record B", "entity_set": entity_set}
      ],
      "stub": True
  }
  ```
- Wrap in `try/except`. On any exception, return `{"error": str(e), "stub": True}`.
- Record `status = "stub"` if `ODATA_ENDPOINT` not set, `status = "success"` or `"error"` otherwise.

**`tool_node.py` responsibility:**
- Only invoke if `state["intent"] == "tool_call"`.
- Extract `entity_set` from `state["prompt"]` using simple regex: first match of `r'entity:\s*(\w+)'`. If no match, default to `"DefaultEntity"`.
- Call `odata_read(entity_set=entity_set)`.
- Append `MCPToolCall` to `state["tool_calls"]`.

## 5. P2-04: FRONTIER FALLBACK

### 5.1 Objective
When `validated == False`, invoke Claude Opus 4.6 with the original prompt + context. Populate `state["fallback_record"]`.

### 5.2 FallbackRecord Schema

```python
class FallbackRecord(TypedDict):
    model: str              # Always "claude-opus-4-6"
    trigger_reason: str     # failure_type from ValidationRecord, or "DIRECT" if intent == fallback_direct
    prompt_sent: str        # Full prompt string sent to Opus
    response: str           # Raw response from Opus
    tokens_used: int        # Total tokens (prompt + completion)
    latency_ms: int         # Wall-clock ms
    timestamp: str          # ISO 8601 UTC
    success: bool           # True if API call completed without exception
```

### 5.3 Prompt Construction for Fallback

```python
prompt_sent = (
    f"[CONTEXT]\n{chr(10).join(c['text'] for c in state['context'])}\n\n"
    f"[QUERY]\n{state['prompt']}\n\n"
    f"[INSTRUCTION]\nAnswer the query using only the provided context. "
    f"Be concise. Do not speculate beyond the context."
)
```

If `state["context"]` is empty, omit the `[CONTEXT]` block.

### 5.4 API Call Spec

- Client: `anthropic.Anthropic()` (reads `ANTHROPIC_API_KEY` from env)
- Model: `"claude-opus-4-6"` (string literal — do not parameterise)
- `max_tokens`: 1024
- `temperature`: 0.2
- Single `user` message with `prompt_sent` as content
- On `anthropic.APIError` or any exception: set `success = False`, `response = str(exception)`, do not raise

### 5.5 After Fallback
- Set `state["fallback_triggered"] = True`
- Set `state["fallback_record"] = <FallbackRecord dict>`
- Set `state["final_response"] = fallback_record["response"]` (used by `response_node`)

## 7. P2-06: PLANNING AGENT

### 7.1 Objective
A self-contained agent that, given any natural language problem prompt, produces a structured breakdown and execution plan. Invoked by `planning_node` when `state["intent"] == "plan"`. Operates within the LangGraph graph — not a standalone process.

### 7.2 Trigger Condition
`intent_node` must classify `intent = "plan"` when the prompt matches any of:
- Starts with: `"plan"`, `"break down"`, `"how do I approach"`, `"design"`, `"architect"`, `"strategy for"`
- Contains phrases: `"step by step"`, `"how would you solve"`, `"what's the approach to"`
- Explicit prefix: `"PLAN:"` anywhere in prompt (case-insensitive)

Classify as `"plan"` if any condition matches. Do not send plan-intent prompts through `local_inference_node`.

### 7.3 Plan Schema

```python
class PlanStep(TypedDict):
    step_id: int                    # 1-indexed
    title: str                      # Short label, max 60 chars
    description: str                # What to do, max 200 chars
    inputs_required: list[str]      # Named inputs this step needs
    outputs_produced: list[str]     # Named outputs this step produces
    depends_on: list[int]           # step_ids this step requires before running
    estimated_complexity: Literal["low", "medium", "high"]
    agent_type: Literal["local", "tool", "frontier", "human"]

class Plan(TypedDict):
    problem_statement: str          # Cleaned restatement of the original prompt
    goal: str                       # Single-sentence desired end state
    constraints: list[str]          # Inferred constraints (empty list if none)
    steps: list[PlanStep]           # Ordered list of steps (min 2, max 10)
    critical_path: list[int]        # Ordered step_ids on the critical (longest dependency) path
    estimated_total_complexity: Literal["low", "medium", "high"]
    created_at: str                 # ISO 8601 UTC
```

### 7.4 Planning Agent Internal Logic

**File:** `agents/planning_agent.py`

**Function signature:**
```python
def run_planning_agent(prompt: str, context: list[dict]) -> Plan:
```

**Processing pipeline (implement in order):**

**Step A — Problem Extraction**
- Strip planning trigger phrases from the front of `prompt` (e.g., remove leading `"plan:"`, `"break down"`, `"PLAN:"`)
- Set `problem_statement` = cleaned prompt

**Step B — Goal Inference**
- Run Phi-3 Mini with this exact system prompt:
  ```
  You are a goal extractor. Given a problem statement, output a single sentence describing the desired end state. Output only the sentence, no preamble.
  ```
- Set `goal` = model output stripped of whitespace
- If output is empty or exceeds 150 chars, set `goal = "Complete: " + problem_statement[:120]`

**Step C — Constraint Detection**
- Run Phi-3 Mini with this exact system prompt:
  ```
  You are a constraint extractor. Given a problem statement, list any constraints mentioned (time, budget, tools, resources, dependencies). Output a JSON array of strings. If none, output []. Output only valid JSON, no preamble.
  ```
- Parse JSON. On parse failure, set `constraints = []`

**Step D — Step Decomposition**
- Run Phi-3 Mini with this exact system prompt (inject `problem_statement` and `goal`):
  ```
  You are a task decomposer. Given a problem and goal, decompose the work into 2–10 concrete steps.
  For each step output a JSON object with keys: title (str), description (str), inputs_required (list[str]), outputs_produced (list[str]), depends_on (list[int], 0 if no deps), estimated_complexity (low|medium|high), agent_type (local|tool|frontier|human).
  Output a JSON array of these objects. Output only valid JSON, no preamble.

  Problem: {problem_statement}
  Goal: {goal}
  ```
- Parse JSON array. Assign `step_id` as 1-indexed position.
- On parse failure or empty array: create a single fallback step:
  ```python
  PlanStep(step_id=1, title="Manual execution required", description=problem_statement[:200],
           inputs_required=[], outputs_produced=["result"], depends_on=[],
           estimated_complexity="high", agent_type="human")
  ```

**Step E — Critical Path Calculation**
- Build dependency graph from `depends_on` lists
- Run longest-path traversal (DAG, topological sort)
- Set `critical_path` = ordered list of `step_id`s on the longest path
- If no dependencies, `critical_path` = all `step_id`s in order

**Step F — Complexity Rollup**
- Count steps by `estimated_complexity`
- If any `"high"` → `estimated_total_complexity = "high"`
- Elif majority `"medium"` → `"medium"`
- Else → `"low"`

**Step G — Assemble & Return**
- Build `Plan` dict from all computed fields
- Set `created_at = datetime.utcnow().isoformat() + "Z"`
- Return `Plan`

### 7.5 `planning_node.py` Responsibility
- Call `run_planning_agent(state["prompt"], state["context"])`
- Set `state["plan"] = result`
- Set `state["final_response"] = json.dumps(state["plan"], indent=2)`
- Append `"planning_node"` to `state["decision_path"]`
- Do NOT invoke `local_inference_node` or `validator_node` for plan-intent prompts

### 7.6 Planning Agent Constraints
- Max 3 Phi-3 Mini calls total (Steps B, C, D) — one call per sub-task, no retries
- Each call: `max_new_tokens=256`, `temperature=0.3`
- Total wall-clock budget: 10 seconds. If exceeded, return whatever partial Plan is assembled, with remaining fields at defaults
