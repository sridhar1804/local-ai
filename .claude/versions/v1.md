# Phase 1 — Phi-3 Mini 4K POC

> **Path note (current layout):** This spec was written when the project had a single `memory/` package containing both runtime trace code and agent-readable docs. Since then, that package was split: Python runtime code (`trace.py`, `sink.py`) lives at [`/model_memory/`](../../model_memory/), and the agent (Claude) knowledge base lives at [`/.claude/memory/`](../memory/). All `memory/trace.py` and `memory/sink.py` references below should be read as `model_memory/trace.py` and `model_memory/sink.py` in the current codebase.

**Mission.** Stand up a single-GPU Phi-3 Mini 4K inference path on WSL, replace the current 7.2 tok/s baseline with a vLLM-backed runtime, and instrument every request with a structured trace. End state: a runnable `python -m main` that takes a query, routes it through a single agent, returns a completion, and writes one JSONL trace per call.

**Out of scope for this phase.** Multi-agent supervision. Tool calling. Retrieval. Validator nodes. Frontier fallback. Quantization beyond AWQ. Concurrent request load testing. Anything that isn't model → router → main + logging.

---

## 0. Diagnosis: 7.2 tok/s Is Broken, Not Slow

A 3090 running Phi-3 Mini (3.8B params) should hit these numbers single-stream:

| Configuration | Expected tok/s |
|---|---|
| HF `transformers` `.generate()` no Flash Attention | 8–30 |
| HF `transformers` + Flash Attention 2 FP16 | 30–60 |
| **vLLM FP16 (target baseline)** | **80–150** |
| vLLM AWQ INT4 | 150–250 |

7.2 tok/s indicates one of these failures:
- HF `transformers` `.generate()` with no Flash Attention, batch size 1, no KV cache reuse
- FP32 weights instead of FP16/BF16
- WSL GPU passthrough partially failing — CPU offload silently kicking in
- llama.cpp with insufficient `n_gpu_layers`

The fix is not optimization, it is a runtime swap. We move to vLLM. Do not pursue quantization until vLLM FP16 clears 80 tok/s — quantization will not fix a broken GPU path.

## Step 1 — Environment Audit

**Goal.** Confirm the GPU pipeline works before installing anything else.

**Output.** A four-line checklist, all green. If any step fails, fix it before continuing — do not install vLLM on a broken CUDA path.

**Suggested steps.**
1. Verify WSL sees the GPU (`nvidia-smi` shows the 3090, driver loaded).
2. Verify CUDA toolchain (`nvcc --version` reports CUDA 12.1+; vLLM requires ≥11.8, prefer 12.x).
3. Verify Python sees CUDA (`torch.cuda.is_available()` returns True; device name matches the 3090).
4. Verify free VRAM ≥ 18 GB. Phi-3 Mini in FP16 needs ~8 GB; we want comfortable headroom.

If `torch.cuda.is_available()` is False, fix WSL passthrough first. Common fixes: update NVIDIA driver on the **Windows host** (not inside WSL), `wsl --update` from PowerShell, reinstall torch from the CUDA 12.1 wheel index.

## Step 3 — Launch vLLM Server

**Goal.** Phi-3 Mini 4K served on port 8000 via vLLM's OpenAI-compatible API.

**Output.**
- `models/server.sh` — single command launches the server with the right config
- Server reachable on `http://localhost:8000/v1/chat/completions`
- Single-stream throughput **≥ 80 tok/s** on a 300-token completion

**Required server config.**
- Model: `microsoft/Phi-3-mini-4k-instruct`
- dtype: float16
- max_model_len: 4096
- gpu_memory_utilization: 0.85
- Disable request logging at the server (we log at the trace layer, not the runtime)
- Configurable via env vars: `MODEL_NAME`, `VLLM_PORT`, `GPU_UTIL`

**Suggested steps.**
1. Write `models/server.sh` with the config above.
2. First run downloads ~7.6 GB of weights to `~/.cache/huggingface/`. Expect 60–90 s load time on a 3090.
3. Smoke test with a curl POST to `/v1/chat/completions`. Confirm a non-empty completion returns.
4. Run a benchmark: send a prompt that produces ~300 completion tokens at temperature 0, measure wall time, divide. Must clear 80 tok/s. If below 60, check `nvidia-smi` during inference — GPU should be at >80% util. **Stop and debug if the floor is not cleared. Do not continue.**

## Step 4 — Trace Schema

**Goal.** A versioned Pydantic schema. One Trace object per request, written as one JSONL line.

**Output.** `memory/trace.py` exporting:
- `SCHEMA_VERSION = "1.0.0"` constant
- `Trace` Pydantic model
- Sub-models: `GenerationRecord`, `ValidationRecord`, `FallbackRecord`

**Required fields on `Trace`.**

| Group | Field | Phase 1 behavior |
|---|---|---|
| Identity | `trace_id` (UUID), `parent_trace_id` (UUID, nullable), `schema_version`, `timestamp_utc` | Auto-generated; `parent_trace_id` null in Phase 1 |
| Input | `input_query`, `session_id`, `user_id` | Query mandatory; session/user nullable |
| Routing | `route`, `decision_path` (list of strings) | Populated as control flow advances |
| Generation | `generation` (GenerationRecord, nullable) | Populated on successful model call |
| Reserved | `retrieval` (dict, nullable), `tool_calls` (list), `validation` (ValidationRecord), `fallback` (FallbackRecord) | **Present but empty/default** — see schema rules |
| Output | `final_output` (string, nullable) | Populated on success |
| Meta | `env` (default `"wsl-dev"`), `git_sha`, `error`, `redaction_applied` (default false) | `error` populated on exception |

**`GenerationRecord` fields.** model, prompt (full assembled, verbatim), completion, prompt_tokens, completion_tokens, latency_ms, sampling (dict).

**`ValidationRecord` fields.** ran (default false), verdict (nullable: pass/fail/soft_pass), confidence (nullable float), failure_type (nullable string).

**`FallbackRecord` fields.** triggered (default false), reason, frontier_model, frontier_completion (all nullable).

**Schema rules — non-negotiable.**
1. `schema_version` is mandatory on every trace.
2. Reserved fields (`retrieval`, `validation`, `fallback`, `tool_calls`) are present in v1.0.0 even though Phase 1 doesn't populate them. This means future analysis code can mix v1.0.0 and v1.1.0 traces without conditional field access.
3. `prompt` stores the **full assembled prompt**, not just the user input.
4. Never mutate a trace after `sink.write()`.
5. The model exposes a `to_jsonl()` method returning a single-line JSON string with no fields excluded.

## Step 5 — Trace Sink

**Goal.** A pluggable interface for writing traces, with a synchronous JSONL implementation for Phase 1.

**Output.** `memory/sink.py` exporting:
- `TraceSink` Protocol — single method `write(trace: Trace) -> None`
- `JsonlTraceSink` class — concrete daily-rotated implementation
- `NullSink` class — for tests

**`JsonlTraceSink` behavior.**
- Constructor takes `log_dir` (default `logs/traces`); creates directory if missing.
- Writes one JSON line per trace to `<log_dir>/<UTC-date>.jsonl`.
- Daily rotation by UTC date — file name reflects the date the trace is written.
- Threadsafe via a single lock. Acceptable for single-process WSL dev; we will revisit when concurrency arrives.
- Append mode only. Never overwrite, never truncate.

**Why an interface.** Future sinks — `KafkaTraceSink`, `ParquetTraceSink`, `RedactingSink` for DPDP compliance — should plug in without changing call sites in `main.py`. A Python `Protocol` (structural typing) keeps this trivial; do not use `ABC`.

**Suggested steps.**
1. Define `TraceSink` Protocol.
2. Implement `JsonlTraceSink` with the behavior above.
3. Implement `NullSink` (write is a no-op).
4. Smoke test: instantiate, write one trace, confirm the file appears under `logs/traces/` and contains valid JSONL.

## Step 6 — Model Client

**Goal.** A thin HTTP wrapper around the vLLM OpenAI-compatible server. No business logic.

**Output.** `models/client.py` exporting:
- `Phi3Client` class
- `GenerationResult` dataclass

**`Phi3Client` contract.**
- Constructor: `base_url` (default `http://localhost:8000/v1`), `model` (default `microsoft/Phi-3-mini-4k-instruct`), `timeout` (default 60 s).
- Single method: `generate(user_message, system_message=None, max_tokens=512, temperature=0.2, top_p=0.95) -> GenerationResult`.
- `close()` method to release the underlying HTTP client.

**`GenerationResult` fields.** completion, prompt_tokens, completion_tokens, latency_ms, model, sampling (dict echoing the params used).

**Hard rules.**
- No retry logic.
- No caching.
- No prompt assembly — that's the agent's job. The client receives a user message and an optional system message, that's it.
- Latency measured at the client wrapping the HTTP call (use `time.perf_counter`).
- Raise on non-2xx HTTP — let the caller decide what to do.

## Step 7 — Main Agent

**Goal.** The only handler in Phase 1. Takes a query, calls the model, returns a result.

**Output.** `agents/main_agent.py` exporting:
- A module-level `SYSTEM_PROMPT` constant
- `AgentResult` dataclass
- `run(query: str, client: Phi3Client) -> AgentResult`

**`AgentResult` fields.** output (the model's completion), assembled_prompt (textual representation of what was sent), generation (the raw `GenerationResult` from the client).

**Responsibilities.**
- Hold the system prompt for this agent.
- Build a textual representation of the assembled prompt for the trace. A simple form like `<system>...</system>\n<user>...</user>` is fine for Phase 1; switch to capturing the literal request payload when tool calling lands in Phase 2.
- Invoke the client.
- Return everything the caller needs to populate a trace.

**Hard rule.** No control flow logic in the agent. No retries, no fallbacks, no validation. Those belong in `main.py` and in Phase 2 nodes. The agent is a function from query to completion plus metadata.

---

## Step 8 — Router

**Goal.** Decide which handler runs. Phase 1 has exactly one route.

**Output.** `agents/router.py` exporting:
- `RouteDecision` dataclass with fields: `route` (string), `reason` (string)
- `route(query: str) -> RouteDecision`

**Phase 1 behavior.** Always returns `RouteDecision(route="main_agent", reason="phase_1_default")`.

**Why build this when it does nothing.** It fixes the call shape in `main.py` so that Phase 2 — validator nodes, multi-agent supervisor, classifier-based routing — extends without refactoring the entry point. The router is a placeholder that earns its keep in the next phase.

## Step 9 — Entry Point (Composition)

**Goal.** `main.py` composes router → agent → trace. This is the canonical control flow that every later phase extends.

**Output.** A runnable `python -m main "your query"` that:
- Prints the completion to stdout
- Writes exactly one JSONL trace under `logs/traces/<UTC-date>.jsonl`
- Returns non-zero exit on failure, **but still writes a trace** with `error` populated

**Top-level flow.**
1. Parse query from `sys.argv[1]`. Print usage and exit 2 if missing.
2. Construct `Phi3Client` and `JsonlTraceSink`.
3. Call a `handle(query, client, sink)` function (described below).
4. Print the returned output to stdout.
5. Always close the client in a `finally` block.

**`handle()` flow.**
1. Construct a `Trace(input_query=query)`.
2. Append `"router.route"` to `trace.decision_path`.
3. Call `router.route(query)`. Set `trace.route`. Append `"router→main_agent"` (or whatever route is returned) to `decision_path`.
4. If route is `main_agent`:
   - Append `"main_agent.run"` to `decision_path`.
   - Call `main_agent.run(query, client)`.
   - Populate `trace.generation` with all fields from the result.
   - Set `trace.final_output`.
   - Append `"done"` to `decision_path`.
   - Return the output.
5. If route is anything else: raise — Phase 1 only knows one route.
6. On any exception:
   - Set `trace.error` to type name + message + traceback.
   - Append `"error"` to `decision_path`.
   - Re-raise.
7. **Always** call `sink.write(trace)` in a `finally` block — non-negotiable.

**Critical invariants.**
- One trace per call, no exceptions.
- Failed calls produce a trace with `error` populated and `decision_path` ending in `"error"`.
- The `finally` block guarantees this. Silent failure is how observability dies.

## Step 10 — End-to-End Smoke Test

**Goal.** Validate happy path and error path.

**Suggested tests.**

**Happy path.** With vLLM server running, run `python -m main "What is the capital of France?"`. Expect:
- A coherent completion printed to stdout
- A new line in `logs/traces/<today>.jsonl`
- The trace, when parsed, contains:
  - `schema_version == "1.0.0"`
  - `decision_path == ["router.route", "router→main_agent", "main_agent.run", "done"]`
  - Populated `generation` block with non-zero `completion_tokens` and `latency_ms`
  - Non-null `final_output`
  - `validation.ran == false`, `fallback.triggered == false`
  - `error == null`

**Error path.** Stop the vLLM server. Run the same command. Expect:
- Non-zero exit code
- A new trace line in the JSONL file with:
  - `error` populated (connection refused or similar)
  - `decision_path` ending in `"error"`
  - `generation == null`, `final_output == null`

If the error case does not produce a trace, the `finally` block in `handle()` is broken. Fix it before moving on.

## Phase 2 Preview (do not build now)

The trace schema is designed to absorb these without breaking changes:
- LangGraph orchestration replacing the hand-written control flow in `main.py`. The `decision_path` simply gets richer; everything else stays the same.
- Validator node populating `ValidationRecord`.
- First MCP-shaped tool (stub OData read), populating `tool_calls`.
- Frontier fallback edge populating `FallbackRecord`.
- AWQ INT4 swap for throughput headroom — only after orchestration is proven, not before.

Each of these depends on the trace shape being stable. Build the substrate first, extend on top.
